# FR-015 — Student Feed Home and Rich Club/Event Discovery

**Priority:** High
**Status:** Planned (students currently land on a dashboard that still shows stats-style cards and chart-oriented sections; the student home should instead be a discovery feed for clubs and events)

---

## Summary

Students should not see KPI-style or analytics-style home content.

Instead, students should land on a feed-first home experience that helps them discover:

- clubs
- published events
- social updates tied to clubs and events

This feature adds a new `Feed` page for students and makes it the effective home page for the student role. Club leaders and admins should keep the current dashboard and KPI-oriented surfaces.

To support a richer feed:

- club pages should become stronger student-facing profile pages
- event pages should become stronger student-facing event pages
- clubs and events should support an optional `x.com` post link that can be embedded in the feed and on detail pages

V1 auth note:

- this plan improves authenticated student discovery surfaces
- it does not require making `/clubs/:id` or `/events/:id` publicly accessible to anonymous users

---

## Goal

Turn the student home experience into a content and discovery surface rather than a management dashboard.

The student home should answer:

- What clubs should I explore?
- What events are coming up?
- What is happening right now on the platform?
- What social post or announcement should I notice?

The feed should encourage browsing, joining clubs, and registering for events.

---

## Role-Based Home Rules

### Student

- student lands on `/feed`
- student does not see KPI charts or analytics widgets on home
- student sees a discovery feed of clubs and published events

### Club leader

- club leader keeps `/dashboard`
- club leader still sees operational dashboard content for their club

### Admin

- admin keeps `/dashboard`
- admin still sees analytics and management-oriented dashboard content

---

## V1 Product Scope

### In scope

- new student `Feed` page
- student role redirected to `Feed` instead of the current dashboard
- feed cards for clubs and published events
- optional X post embed on clubs and events
- improved club detail page
- improved event detail page

### Out of scope for V1

- full recommendation engine
- personalized ranking by ML
- comments or reactions on feed items
- multiple embedded posts per club/event
- full social posting system inside CMP

---

## Product Rules

### Student home rules

- students must not see dashboard charts on their home surface
- only public-safe student content appears in the feed
- only `published` events appear in the student feed
- V1 keeps club and event detail routes authenticated; “public-safe” here means appropriate for student discovery, not anonymous internet access

### Feed rules

- feed may contain both `club` items and `event` items
- cards must link directly to club and event detail pages
- club cards should highlight identity, description, and next activity
- event cards should highlight date, location or mode, and CTA state

### X embed rules

- each club may store one optional primary X post URL
- each event may store one optional primary X post URL
- only valid X/Twitter status URLs should be accepted
- if an embed fails, the UI must fall back to a normal link instead of breaking the page

### Recommended endpoint

```txt
GET /api/feed?limit=20&offset=0
```

### Recommended response shape

```json
{
  "items": [
    {
      "type": "event",
      "id": 15,
      "title": "AI Workshop",
      "club": {
        "id": 2,
        "name": "AI Club",
        "logo_url": "/uploads/logos/club-2.png"
      },
      "description": "Hands-on workshop about practical AI",
      "starts_at": "2026-04-20T16:00:00Z",
      "location": "Lab 3",
      "delivery_mode": "physical",
      "members_only": false,
      "registration_status": "not_registered",
      "x_post_url": "https://x.com/example/status/123"
    },
    {
      "type": "club",
      "id": 2,
      "name": "AI Club",
      "description": "Building practical AI projects",
      "logo_url": "/uploads/logos/club-2.png",
      "membership_status": "none",
      "next_event": {
        "id": 15,
        "title": "AI Workshop",
        "starts_at": "2026-04-20T16:00:00Z",
        "delivery_mode": "physical"
      },
      "x_post_url": "https://x.com/example/status/123"
    }
  ],
  "total": 20
}
```

### Feed composition rules

- event items:
  - only `published`
  - prioritize upcoming events
- club items:
  - include public club info
  - include next upcoming published event when available
- interleave club and event items so the page feels like a feed, not just two separate lists
- include enough user-specific CTA state in each item to avoid N+1 fetches just to render feed actions
  - club items should include membership/request state
  - event items should include registration state plus `members_only` and `delivery_mode`

## Assumptions

- Students should have a discovery-first home, not a management dashboard.
- Existing KPI and analytics surfaces remain valuable for admins and club leaders.
- V1 keeps club and event detail pages behind authenticated routes; this plan is about better student discovery UX, not anonymous-public page delivery.
- One primary X post per club or event is enough for V1.
- Feed items should carry enough CTA state for the page to render join/register actions without additional per-card API calls.
- If an X embed cannot be rendered in a given environment, the system should fall back to a plain external link rather than failing the page.
