# FR-012 — XP and Levels System

**Priority:** High
**Status:** Planned (the app already tracks attendance, achievements, memberships, and login activity, but it does not yet convert that activity into a student-facing XP/level system)

---

## Summary

Replace the current plain idea of "student progress" with a gamified progression layer.

Students earn XP (Experience Points) when they use the platform and participate in club life. As XP increases, the student levels up and sees a visible progress bar toward the next level.

This should become the main student-facing progress model, while existing KPI and achievement logic can still remain for admin/club reporting.

---

## Goal

Make student progress feel rewarding and easy to understand by showing:

- Current XP
- Current level
- Progress to next level
- Recent XP activity history
- Level-up feedback when new milestones are reached

This feature should increase retention, repeat logins, event attendance, club participation, and profile completion.

---

## V1 Product Scope

### Student actions that award XP

Implement XP for actions that already fit the current codebase first:

| Action | V1 status | Notes |
|---|---|---|
| Join a club | In scope | Award when membership becomes `active`, not when request is still `pending` |
| Attend an event | In scope | Award on successful attendance check-in |
| Log in regularly | In scope | Award once per day through existing `user_login_activity` flow |
| Complete profile | In scope | One-time award when required profile fields are complete |
| Participate in club activities | Partial | Treat verified attendance as the first supported activity source |
| Complete weekly tasks | Future-ready | Hook supported by XP engine, but task feature does not exist yet |
| Answer daily quiz questions | Future-ready | Hook supported by XP engine, but quiz feature does not exist yet |

### Recommended default XP values

| Action key | XP |
|---|---:|
| `membership_joined` | 40 |
| `event_attended` | 25 |
| `daily_login` | 5 |
| `profile_completed` | 20 |
| `weekly_task_completed` | 15 |
| `daily_quiz_answered` | 10 |
| `club_activity_participated` | 20 |

These values should live in a centralized backend config/service so they can be adjusted later without rewriting business logic.

---

## Level Model

Use explicit level thresholds in V1 instead of a complex formula.

| Level | XP range |
|---|---|
| 1 | 0-99 |
| 2 | 100-249 |
| 3 | 250-499 |
| 4 | 500-799 |
| 5 | 800-1199 |
| 6 | 1200-1699 |

Backend should calculate:

- `current_level`
- `current_xp`
- `current_level_floor`
- `next_level_xp`
- `xp_to_next_level`
- `progress_percent`

Example:

```ts
progress_percent =
  ((current_xp - current_level_floor) / (next_level_xp - current_level_floor)) * 100;
```

## Step 1 — Data Model and Migration

Create a new migration:

**File:** `apps/backend/migrations/025_create_student_xp_system.sql`

### Add user XP summary fields

Add cached fields to `users`:

- `xp_total INTEGER NOT NULL DEFAULT 0`
- `current_level INTEGER NOT NULL DEFAULT 1`
- `profile_completed_at TEXT NULL`

### Add XP transaction ledger

Create `xp_transactions`:

```sql
CREATE TABLE xp_transactions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER NOT NULL,
  action_key TEXT NOT NULL,
  xp_delta INTEGER NOT NULL,
  source_type TEXT NULL,
  source_id INTEGER NULL,
  reference_key TEXT NOT NULL UNIQUE,
  metadata_json TEXT NULL,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

Why both summary fields and a ledger:

- `xp_transactions` is the source of truth and audit trail
- `users.xp_total` and `users.current_level` make reads fast for profile/dashboard UI

## Step 2 — Backend XP Service

Create:

**File:** `apps/backend/src/services/gamification.service.ts`

This service should own:

- XP rule map
- level-threshold map
- `calculateLevel(xpTotal)`
- `getLevelProgress(xpTotal)`
- `awardXp(...)`
- `rebuildUserXp(userId)` for backfill or repair

Recommended `awardXp(...)` behavior:

1. Build a deterministic `reference_key`
2. Insert an `xp_transactions` row only if that reference is new
3. Recompute `xp_total`
4. Recompute `current_level`
5. Return whether a level-up happened

Example reference keys:

- `login:USER_ID:YYYY-MM-DD`
- `attendance:EVENT_ID:USER_ID`
- `membership:CLUB_ID:USER_ID`
- `profile:USER_ID`

This prevents duplicate XP when the same action is retried.

## Step 3 — Hook XP into Existing Backend Flows

### 3a. Daily login XP

**File:** `apps/backend/src/controllers/users.controller.ts`

Extend `recordLoginActivity`:

- keep existing `user_login_activity` insert
- if the insert created a new day entry, award `daily_login` XP
- return:
  - `date`
  - `xp_awarded`
  - `new_unlocks`
  - `level_up`
  - updated gamification summary

### 3b. Club join XP

**File:** `apps/backend/src/controllers/membership.controller.ts`

Award XP only when a membership changes to `active`.

Reason:

- `joinClub()` currently creates `pending` requests
- rewarding at `pending` would be easy to abuse
- rewarding at approval means the student truly joined the club

### 3c. Event attendance XP

**File:** `apps/backend/src/controllers/attendance.controller.ts`

After successful check-in:

- award `event_attended` XP
- use one XP record per `(event_id, user_id)`
- support both QR and manual check-in without double-awarding

### 3d. Profile completion XP

**Files:**

- `apps/backend/src/controllers/users.controller.ts`
- `apps/backend/src/models/user.model.ts`

When `updateMe` makes the profile complete for the first time:

- set `profile_completed_at`
- award one-time `profile_completed` XP

For V1, define profile completion using fields the app already has:

- non-empty `name`
- non-empty `avatar_url`

If the profile schema grows later, this rule can be expanded.

## Step 7 — Backfill and Migration Safety

Existing students should not all start at zero if the platform already has participation history.

Create a one-time rebuild path that can derive XP from historical data:

- active memberships -> `membership_joined`
- attendance rows -> `event_attended`
- `user_login_activity` rows -> `daily_login`
- already-complete profiles -> `profile_completed`

Recommended implementation:

- add `rebuildUserXp(userId)` in `gamification.service.ts`
- optionally add an admin-only script:
  - `apps/backend/src/scripts/backfill-gamification.ts`

The rebuild must be idempotent and respect `reference_key` uniqueness.

---

## Anti-Abuse Rules

- Daily login XP only once per calendar day
- Event attendance XP only once per event
- Membership XP only once per club activation
- Profile completion XP only once per account
- Future quiz/task hooks must also use deterministic `reference_key` values

---

## Acceptance Criteria Checklist

- [ ] Each student has `current_xp`, `current_level`, and a valid next-level progress calculation
- [ ] Successful attendance check-in awards XP exactly once per event
- [ ] Membership XP is awarded only when membership becomes `active`
- [ ] Daily login XP is awarded at most once per day
- [ ] Profile completion XP is awarded only once
- [ ] Level thresholds match the configured ranges
- [ ] Student dashboard and profile page both show XP and level progress
- [ ] Recent XP history is visible to the student
- [ ] Duplicate retries do not create duplicate XP transactions
- [ ] Existing student activity can be backfilled safely

## Assumptions

- "Joining a club" means becoming an approved active member, not just submitting a request.
- Existing KPI and achievement logic should remain for operational reporting; FR012 changes the student-facing progress experience.
- Weekly tasks and daily quizzes are not yet implemented in this codebase, so FR012 should provide reusable XP hooks for those future features rather than blocking on them now.
- V1 should focus on actions that already exist in the platform and can be verified reliably.
