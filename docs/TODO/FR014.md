# FR-014 — Replace Achievements with a Badge System

**Priority:** High
**Status:** Planned (the app already has manual achievements plus an automatic achievement engine, but the product language and UX are inconsistent; badges should become the single student-facing recognition system)

---

## Summary

Replace the current student-facing achievements experience with a clearer badge system.

Students should earn badges automatically when they hit important milestones or complete specific activities.

Examples:

- `First Step` — joined the first club
- `Explorer` — visited 5 club pages
- `Active Participant` — attended 3 events
- `Quiz Master` — answered 10 questions correctly
- `Weekly Hero` — completed all weekly missions

Badges should appear on the student profile and one selected badge can also be shown next to the student’s name in selected parts of the platform.

---

## Why This Is a Replacement, Not Just a New Feature

The current codebase already contains two different recognition concepts:

1. A manual `achievements` table that club leaders/admins can create and remove
2. An automatic engine using `achievement_definitions` and `achievement_unlocks`

That split makes the product confusing. FR014 should standardize the experience so:

- badges become the main recognition model for students
- milestone recognition is automatic by default
- the UI talks about `badges`, not `achievements`
- legacy manual achievement records are treated as compatibility data, not the primary system

---

## Goal

Create a recognition system that is:

- visible
- motivating
- automatic
- easy to understand
- aligned with the gamification work in FR012 and daily MCQs in FR013

The badge system should reward milestone progress, while FR012 continues to represent continuous XP/level progress.

---

## V1 Product Scope

### In scope

- Student badges awarded automatically from tracked platform activity
- Badge catalog with seeded milestone definitions
- Badge display on profile
- One featured badge shown next to the student’s name in selected UI areas
- Rename student-facing achievements UX to badges
- Compatibility path from current achievement engine to badge engine

### Out of scope for V1

- Manual badge design studio for club leaders
- Uploading custom badge artwork
- Club-specific custom badge rules
- Badge trading or gifting
- Badge expiration

---

## Product Rules

### Awarding rules

- Badges are awarded automatically when conditions are met
- A badge can only be unlocked once per student
- Unlocking should be deterministic and idempotent
- Students do not manually apply for badges

### Display rules

- All unlocked badges appear on the student profile
- A student may select one unlocked badge as their featured badge
- Only unlocked badges may be featured

### Replacement rules

- Student-facing pages and navigation should say `Badges`
- Legacy manual achievements should no longer be the default recognition surface
- Existing achievement data should not be deleted blindly; it should be preserved or migrated safely

## Step 1 — Data Model and Migration

Create the next migration after FR013, for example:

**File:** `apps/backend/migrations/027_replace_achievements_with_badges.sql`

### Create `badge_definitions`

```sql
CREATE TABLE badge_definitions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  code TEXT NOT NULL UNIQUE,
  entity_type TEXT NOT NULL DEFAULT 'student',
  name TEXT NOT NULL,
  name_ar TEXT NOT NULL,
  description TEXT NOT NULL,
  description_ar TEXT NOT NULL,
  icon_key TEXT NOT NULL,
  rarity TEXT NOT NULL DEFAULT 'common',
  metric_key TEXT NOT NULL,
  threshold INTEGER NOT NULL,
  is_active INTEGER NOT NULL DEFAULT 1,
  sort_order INTEGER NOT NULL DEFAULT 0
);
```

### Create `badge_unlocks`

```sql
CREATE TABLE badge_unlocks (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  badge_definition_id INTEGER NOT NULL,
  user_id INTEGER NOT NULL,
  unlocked_at TEXT NOT NULL DEFAULT (datetime('now')),
  source_type TEXT NULL,
  source_id INTEGER NULL,
  FOREIGN KEY (badge_definition_id) REFERENCES badge_definitions(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
  UNIQUE(badge_definition_id, user_id)
);
```

### Add featured badge field to `users`

Add:

- `featured_badge_definition_id INTEGER NULL`

This enables badge display next to the student name without forcing every unlocked badge to show everywhere.

### Migration strategy

To avoid breaking the current app immediately:

1. Seed `badge_definitions` from the new badge catalog
2. Copy compatible rows from `achievement_definitions` into `badge_definitions` where useful
3. Copy existing student `achievement_unlocks` into `badge_unlocks` where codes still map cleanly
4. Keep the legacy `achievements` table intact during transition

This should be a safe migration, not a destructive one.

## Step 2 — Backend Badge Engine

Create:

- `apps/backend/src/models/badge.model.ts`
- `apps/backend/src/services/badge-engine.service.ts`

This should be implemented as an evolution of the current `achievement-engine.service.ts`, not as a disconnected second engine that duplicates the same metric logic.

Recommended migration approach:

- refactor the existing achievement engine into badge-oriented behavior where possible
- keep a thin compatibility layer for any legacy code paths that still call achievement progress APIs during transition
- only introduce a brand-new service file if it clearly wraps or succeeds the existing engine rather than re-implementing it from scratch

The badge engine should own:

- badge definition reads
- metric calculation
- badge evaluation
- unlock creation
- featured badge validation
- progress payloads for the frontend

Recommended service methods:

- `getStudentBadgeMetrics(userId)`
- `evaluateStudentBadges(userId)`
- `listUnlockedBadges(userId)`
- `listBadgeCatalog()`
- `setFeaturedBadge(userId, badgeDefinitionId)`
- `getStudentBadgeProgress(userId)`

## Step 3 — Badge Metrics

Badge metrics should be derived from real platform data already present in the codebase.

### Core V1 metrics

| Metric key | How to compute |
|---|---|
| `joined_clubs_count` | count active memberships for the user |
| `distinct_club_pages_visited` | count distinct `/clubs/:id` page views for the authenticated user |
| `attendance_count` | count attendance rows for the user |
| `daily_questions_correct_count` | count correct daily-question answers |
| `weekly_missions_completed_count` | reserved for future feature |

### Existing data sources

- `memberships`
- `attendance`
- `page_views`
- `daily_question_answers` from FR013

For `Explorer`, use distinct club IDs parsed from page-view paths such as:

```txt
/clubs/1
/clubs/7
/clubs/12
```

Only authenticated page views should count toward badge progress.

Recommended `GET /api/badges/me` response:

```json
  "unlocked": [
    {
      "badge_definition_id": 3,
      "code": "attendance_3",
      "name": "Active Participant",
      "rarity": "common",
      "unlocked_at": "2026-04-16T09:10:00Z"
    }
  ]
}
```

### Compatibility strategy

During transition:

- keep `/api/achievements/engine/progress/me` as a temporary compatibility alias if needed
- keep `/api/achievements/engine/progress/club/:clubId` as a temporary compatibility alias if needed
- keep legacy achievement-report and manual-achievement endpoints available for admin/club-leader historical workflows until those flows are deliberately migrated
- make new frontend work against `/api/badges/*`
- reduce use of manual `POST /api/achievements` and `DELETE /api/achievements/:id`

Manual achievement creation should no longer be the primary student-recognition workflow.

## Step 7 — Legacy Achievements Handling

The existing `achievements` table should become legacy data.

Recommended policy:

- do not delete old rows automatically
- hide manual achievements from the default student recognition flow
- keep legacy records accessible only where needed for compatibility

### Important impact

FR-001 is currently written around achievement reports. FR014 changes the recognition model, so the report feature should eventually become one of:

1. a badge report
2. a broader recognition report
3. a legacy achievement report kept only for historical records

That report migration is related work and should be tracked explicitly if adopted.

---

## Step 8 — Admin Catalog Management

V1 can seed the initial badge catalog via migration, but admins should eventually have limited badge-catalog controls.

Optional admin actions:

- enable/disable a badge definition
- reorder catalog display
- adjust icon or rarity metadata

Do not allow arbitrary admin awarding in V1 unless there is a strong business reason.

Automatic rules are the core of the badge system.

## Assumptions

- FR014 replaces the student-facing recognition language and UX, but should migrate legacy data safely rather than deleting it.
- FR012 remains the continuous progression system for XP and levels; FR014 is the milestone-recognition layer.
- The current `achievement-engine.service.ts` should be refactored or wrapped during migration rather than abandoned abruptly, because current routes and pages already depend on it.
- FR013 provides the data source needed for badges like `Quiz Master`.
- V1 guarantees featured-badge display on the profile header; leaderboard/member-row display is a follow-up unless those APIs are explicitly extended.
- Weekly missions do not exist yet in this codebase, so `Weekly Hero` should be treated as future-ready rather than blocking FR014.
